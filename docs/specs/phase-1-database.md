## 1. Objective

Give the app **durable, per-user, server-side persistence**. After Phase 1, a
signed-in user's identity and books live in Postgres (not a browser), scoped to
their account, surviving cache clears and working across devices. This is the
keystone every later phase builds on.

**Out of scope (deferred):** profiles/onboarding (Phase 2), autosave/history
(Phase 3), billing (Phase 5), ingestion (Phase 6). We persist what exists today;
we do not add new product surface.

---

## 2. Guiding principles (three lenses)

**Principal AI engineer**
- AI server actions stay deterministic-fallback safe; persistence wraps them, never blocks them.
- Every AI generation is attributable to a `userId` + `projectId` (sets up metering in Phase 5).
- Idempotent writes: regenerating a blueprint replaces, never duplicates.

**Data engineer**
- Normalized relational model; foreign keys + `ON DELETE CASCADE`; explicit indexes on every FK and query path.
- Migrations are **versioned SQL files committed to the repo** (`generate` + `migrate`), never ad-hoc `push` against prod.
- Serverless-safe connection handling (bounded pool; one client per module instance).
- Timestamps (`created_at`/`updated_at`) on every table; UTC; app-managed `updated_at`.

**DB / ops admin**
- Least-privilege: app connects as a non-superuser role.
- All secrets in Vercel/Railway env, never in the repo.
- Authorization enforced in the data layer: **every query filtered by `userId`** (no implicit "all rows").
- Backups: rely on Railway automated backups; document restore.
- Observability: log slow/failed queries; surface DB errors to Sentry (seeded here, matured Phase 5).

## 3. Architecture Decision Records

**ADR-1 · Session strategy: JWT + Drizzle adapter.**
Keep `strategy: "jwt"` AND attach the Auth.js Drizzle adapter. Adapter persists
durable identity (`users`, `accounts`); JWT keeps middleware edge-fast (no DB
hit per request). Inject `userId` into the token (`jwt` callback) and expose it
on `session.user.id` (`session` callback). _Rejected:_ database sessions (extra
DB round-trip on every request, no benefit yet).

**ADR-2 · Driver & pooling: `postgres-js` via `drizzle-orm/postgres-js`, bounded pool.**
Instantiate a single client per server module with a small `max` (e.g. 1–5) and
`idle_timeout`, to avoid exhausting Postgres connections under serverless
fan-out. _Risk:_ Railway Postgres has no built-in pooler like Neon; if we hit
connection limits we add PgBouncer or migrate to a pooled provider. Documented
in Risks.

**ADR-3 · Migrations: `drizzle-kit generate` → commit SQL → `drizzle-kit migrate`.**
Migration files are reviewed artifacts in `/drizzle`. `db:push` allowed only for
local throwaway iteration. **Production applies via a serialized CI step**
(`drizzle-kit migrate` in the deploy workflow) that runs BEFORE deploy — never
at app boot (serverless instances would race). Additive/expand-contract
migrations make "schema ahead of code" the safe ordering. The step is guarded
on `DATABASE_URL` so it can't block deploys before the DB exists; a manual
`workflow_dispatch` baselines the schema once before the data cutover lands.

**ADR-4 · Data access: server-only repository + server actions; RSC for reads.**
A `src/lib/data/*` layer (server-only, pure functions over Drizzle) is the single
place that touches the DB. Server Actions wrap it for mutations; Server
Components read through it. Client components call actions and revalidate.
_Keeps the exact operations the Zustand store exposes_ (`createProject`,
`getProject`, `patchChapter`, `approveBlueprint`, …) so call sites change
minimally.

**ADR-5 · IDs: keep app-generated `text` ids for domain rows; adapter owns user ids.**
`book_projects.id` etc. stay prefixed `text` ids (`uid("proj")`) for stable URLs.
`users.id` is generated by the adapter (UUID). FKs are `text`.

**ADR-6 · Authorization: app-layer `userId` scoping now; Postgres RLS later.**
Every repository read/write takes `userId` and filters by it; a row that doesn't
belong to the caller is a 404, never returned. RLS is noted as future
defense-in-depth (needs per-request role/session vars).

**ADR-7 · Cutover: full replace of localStorage persistence + one-time import.**
No dual-write (no real data to protect — see ROADMAP). On first authenticated
load, if a localStorage project/draft exists, offer a one-time import to the
account, then clear it. Ephemeral UI state may remain client-side.

## 4. Data model (Phase 1 scope)

Builds on `src/lib/db/schema.ts`. Changes: add Auth.js adapter tables, add
`user_id` FKs + relations + indexes, drop legacy `clerk_user_id`.

| Table | Notes |
|-------|-------|
| `users` | Adapter-owned: `id`, `name`, `email`, `email_verified`, `image`. **Drop `clerk_user_id`.** |
| `accounts` | Adapter: OAuth account links (provider, providerAccountId, tokens). FK → users, cascade. |
| `sessions` | Adapter table (present for compatibility even under JWT). |
| `verification_tokens` | Adapter table. |
| `book_projects` | Add `user_id` FK → `users.id` (cascade). Index `(user_id, updated_at)`. |
| `book_blueprints` | FK → `book_projects.id` cascade. Index `(project_id)`. |
| `chapters` | FK → project cascade. Index `(project_id, order_index)`. |
| `source_content` | FK → project cascade. Index `(project_id)`. |
| `publishing_kits` | FK → project cascade. Unique `(project_id)`. |

Constraints: NOT NULL on required cols; `updated_at` set by the repo on every
write; `ON DELETE CASCADE` so deleting a project removes its children.

## 5. Features (epics)

- **E1 — DB connectivity & schema:** Drizzle client, finalized schema, committed migrations.
- **E2 — Identity persistence:** Auth.js Drizzle adapter; `userId` in session; legacy Clerk column removed.
- **E3 — Book data-access layer:** server-only repository + server actions mirroring the store contract.
- **E4 — Surface migration:** dashboard, builder (create/approve), project workspace, publishing read/write from DB.
- **E5 — One-time localStorage import:** migrate an in-flight local project to the account on first login.
- **E6 — Hardening:** indexes, `userId` scoping, error handling, Sentry seed, tests.

## 6. User stories (with acceptance criteria)

**US-1 (E2) — Returning user, same account.**
_As a signed-in user, I want my account to be remembered so my books are always mine._
- AC: second login with the same Google account resolves to the same `users.id`.
- AC: `session.user.id` is populated on server and client.

**US-2 (E4) — Persisted book creation.**
_As a user, when I approve a blueprint, I want the book saved to my account._
- AC: approving writes a `book_projects` row (+ blueprint + chapters) with my `user_id`.
- AC: the project opens at `/project/[id]` reading from the DB.

**US-3 (E4) — Cross-device dashboard.**
_As a user, I want to see my books on any device._
- AC: `/dashboard` lists only my projects, newest first, fetched server-side.
- AC: logging in on another browser shows the same list.

**US-4 (E4) — Edit persistence.**
_As a user, edits to chapters/blueprint/publishing kit persist._
- AC: chapter content, status, and publishing-kit fields survive reload.
- AC: `updated_at` advances on each write.

**US-5 (E6) — Isolation.**
_As a user, I can never see or modify another user's book._
- AC: requesting a `projectId` I don't own returns 404 (not the row).
- AC: every repository method requires and filters by `userId`.

**US-6 (E5) — No lost work at cutover.**
_As an existing visitor mid-builder, I don't lose my local draft when I sign in._
- AC: on first authenticated load with a local project present, I'm offered a one-time import; on accept it's saved to my account and local copy cleared.

**US-7 (E1, admin) — Reproducible schema.**
_As an engineer, I can recreate the schema from committed migrations._
- AC: `drizzle-kit migrate` on a fresh DB yields the exact schema.
- AC: migrations are committed under `/drizzle`.

## 7. Engineering task specs (ordered)

> Each task: small, independently reviewable, typecheck + tests green before merge.

1. **T1 — Install + DB client.** Add `postgres`, `@auth/drizzle-adapter`. Create
   `src/lib/db/client.ts` (bounded-pool `postgres-js` + `drizzle`), server-only.
   _DoD:_ importable in a server action; no client bundle leakage.
2. **T2 — Finalize schema.** Add adapter tables, `user_id` FKs, relations,
   indexes; drop `clerk_user_id`. _DoD:_ `drizzle-kit generate` produces a clean migration.
3. **T3 — Commit migration.** Generate + commit SQL to `/drizzle`. _DoD:_ migrate runs clean on a scratch DB.
4. **T4 — Wire adapter + session.** Add adapter to `src/auth.ts`; `jwt`/`session`
   callbacks inject `userId`. _DoD:_ `session.user.id` present; sign-in persists a `users` row.
5. **T5 — Repository layer.** `src/lib/data/projects.ts` (+ chapters, blueprint,
   publishing) — server-only functions, all `userId`-scoped, mirroring store ops.
   _DoD:_ unit tests for each, including isolation (wrong-user → null).
6. **T6 — Server actions.** Thin `"use server"` wrappers + `revalidatePath`.
   _DoD:_ callable from client; auth-guarded (reject unauthenticated).
7. **T7 — Migrate dashboard.** `/dashboard` → Server Component reading the repo. _DoD:_ US-3 AC.
8. **T8 — Migrate builder.** `approve()` calls the create action; route to DB project. _DoD:_ US-2 AC.
9. **T9 — Migrate workspace + publishing.** `/project/[id]` reads/writes via actions. _DoD:_ US-4 AC.
10. **T10 — One-time import.** Client check on first authed load → import action → clear local. _DoD:_ US-6 AC.
11. **T11 — Hardening.** Error handling, not-found → 404, Sentry seed, slow-query log. _DoD:_ US-5 AC; tests green.

## 8. Non-functional requirements

- **Security:** no secret in repo; least-privilege DB role; `userId` scoping on 100% of queries; server-only data layer (`import "server-only"`).
- **Performance:** indexed FKs + list query; bounded connection pool; reads via RSC (no client waterfalls).
- **Reliability:** cascade deletes; idempotent blueprint replace; migrations reversible/forward-only and committed.
- **Privacy:** store only what we use; deletion path designed (full delete lands with Account/Settings in Phase 2/5).

---

## 9. Testing strategy

- **Unit:** repository functions (CRUD + isolation) against a test DB or in-memory pg; keep existing 76 tests green.
- **Integration:** create → approve → read project round-trip via actions.
- **Manual (preview→prod):** sign in, approve blueprint, reload, second-device check, wrong-`projectId` → 404.
- **Gate:** typecheck → tests → Vercel build, as today.

---

## 10. Risks & mitigations

| Risk | Mitigation |
|------|------------|
| Serverless connection exhaustion (Railway has no built-in pooler) | Bounded `max`, `idle_timeout`; add PgBouncer / pooled provider if limits hit |
| Client-heavy components → async refactor churn | Repository seam keeps store ops identical; migrate one surface per task (T7–T9) |
| `server-only` data layer leaking to client bundle | `import "server-only"`; actions are the only client-facing entry |
| Migration applied to prod by hand | Documented runbook; automate in CI in a later phase |
